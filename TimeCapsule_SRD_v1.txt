1.1  Document Purpose
This System Requirements Document (SRD) defines all functional requirements, non-functional requirements, data model specifications, interface requirements, security rules, Firestore index configuration, system constraints, test requirements, and open issues for the Time Capsule web application.
It is the authoritative reference for the Antigravity development team. Every feature built must trace back to a requirement stated in this document. No feature is implemented that does not appear here without a formal change request.

1.2  Scope
This document covers Phase 1 (MVP) of the Time Capsule product — a fully serverless web application built on React, Firebase, and Vercel, deployed initially as a pilot at a single engineering college in India.
Phase 2 and Phase 3 features (multi-college support, 3D campus map, AR, SaaS whitelabel, admin dashboard) are out of scope for this document and are explicitly listed in Section 11.

1.4  Definitions & Abbreviations



2.1  Product Description
Time Capsule is a web application that enables final-year engineering students to preserve campus knowledge for future batches. Seniors post tips covering academics, social life, infrastructure, and career. Juniors search and access this knowledge instantly, filtered by department, category, and campus location.
The platform is designed for zero-friction access. Any student with a valid institutional email can log in and immediately access all content. There are no contribution walls, no paywalls, and no locked features. The platform runs entirely in a browser — no installation, no download.

2.3  System Context
Time Capsule operates entirely within the browser. It communicates directly with Firebase services via the Firebase JavaScript SDK. There are no intermediate API servers. All business logic that can be enforced server-side is enforced via Firestore Security Rules.
Students access the application via any modern browser on mobile or desktop
Firebase Auth handles all authentication and session management
Firestore handles all data storage and real-time updates
Vercel serves the static React build globally via CDN

10.1  Hard Constraints


10.2  Assumptions

The following assumptions are made in this document. If any assumption proves incorrect, a formal change request is required.
The pilot college has an institutional Google Workspace account issuing @college.edu.in style email addresses that work with Google OAuth.
Students at the pilot college have smartphones with modern browsers capable of running a React SPA.
The Firebase free tier (Spark plan) is sufficient for the Phase 1 pilot of 500 MAU. This assumption must be revisited before Phase 2.
The college does not require a custom domain for Phase 1. A Vercel subdomain is acceptable.
The founding team acts as platform admin for Phase 1. No college representative admin is required until Phase 2.
Content moderation is handled entirely by the three-layer system (verified identity + community flagging + admin deletion). No AI moderation is required in Phase 1.
Semester dates are well-defined and consistent enough that the auto-applied semester stamp (based on the calendar date of posting) is accurate.
