Institutional Memory Resets Every Year
Every year at every engineering college in India, final-year students graduate and take four years of hard-earned campus knowledge with them — permanently. The next batch arrives and starts from zero. Again.

This is not a single-college problem. It is a full institutional memory reset, every single year, at every one of the 4,000+ engineering colleges in India.

What Gets Lost Every Year
Which professor actually responds to emails and which does not
Which labs to avoid and which ones have the equipment that actually works
Where to print assignments at 11pm before a submission deadline
How placements actually work — what really happens vs what orientation says
Which electives are genuinely worth taking
Unwritten rules of the campus — every college has them, nobody writes them down

Why Existing Solutions Fail



Time Capsule — A Living Knowledge Base for Campus Life
Time Capsule is a web application where final-year students leave campus knowledge behind in the form of structured tips before they graduate. Juniors can search, filter, and find these tips instantly — by department, location, and category.


How It Works — Three Steps



1. Institutional Email Login
The very first thing a user sees is the login screen. Access is restricted exclusively to the college's institutional email ID. One tap using the Google account students already have. No new password. No signup friction. No outsiders.
Eliminates anonymous abuse before it starts — no anonymity means no garbage posts
Makes the platform feel like it belongs to the institution, not a generic social network
Automatically restricts the community to verified members of that college

2. Completely Open Access — No Content Gates
All tips posted by seniors are visible to every logged-in student immediately. There is no requirement to post before you can read. There is no contribution wall. There is no locked content.
Students love free. Free and open access drives adoption.
Reading is a gateway behaviour — students who read tips naturally become invested in the platform and many will post voluntarily
Contribution comes from engagement, not coercion — a well-designed platform creates voluntary contributors


3. Four Categories
Every tip belongs to exactly one of four categories:
Academics — Professors, courses, study strategies, exam tips, electives
Social — Events, clubs, campus culture, unwritten social rules
Infrastructure — Facilities, labs, hostels, canteen, printing, transport
Career — Placements, internships, companies, resume tips, interview prep

4. Department Filter
Students filter tips by their branch — CSE, ECE, Mechanical, Civil, and so on. A CSE junior only sees CSE-relevant tips by default. This removes noise and makes every result feel personally relevant.

5. Location Tags
Every tip is tagged to a campus location — library, lab, canteen, hostel, department building. Students can browse by location and discover tips relevant to where they physically are.

6. Upvote System
Students upvote tips they find genuinely useful. The most helpful tips rise to the top. Bad or outdated content gets ignored naturally. No complex algorithm — community votes surface the best knowledge.

7. Semester Stamp
Every tip is automatically stamped with the semester and year it was posted. Students can judge relevance themselves without needing an admin to curate or expire content manually.



What Gets Built in Phase 1
Every feature below must ship before the platform goes live at the pilot college. No feature is decorative.


What Is NOT in MVP
No native mobile app — the web application works in any mobile browser
No 3D campus map — deferred to Phase 2
No AR or geofencing features — deferred to Phase 2
No AI recommendations — deferred after adoption is proven
No complex gamification — upvotes are the only mechanic
No dedicated moderation dashboard — admin handles via Firebase console




Step-by-Step Implementation Sequence
The following is the exact order in which features should be designed, built, and tested. Each step is a discrete deliverable. Nothing begins until the previous step is reviewed and approved.

STEP 1 — Project Setup & Infrastructure
Initialise React + Vite project
Install Tailwind CSS and configure with the brand design system
Connect Firebase project (Auth, Firestore, Storage)
Configure Google OAuth — restrict to institutional email domain
Deploy skeleton to Vercel with CI/CD from GitHub
Verify: login with institutional email works end-to-end

STEP 2 — Authentication Flow
Build login screen — hero landing with tagline, institutional email prompt
Implement Google OAuth one-tap sign-in
Implement domain validation (reject non-institutional emails at auth layer)
On first login, capture department and year via onboarding screen
Create user document in Firestore on first login
Verify: new user can sign in, profile is created, domain validation rejects outsiders

STEP 3 — Tip Creation
Build the Post a Tip screen
Form fields: title (max 120 chars), content (max 2,000 chars), category selector, department selector, location tag
Semester stamp auto-applied on submission (no user input required)
Write tip to Firestore /tips/ collection
Verify: tip appears in Firestore with all fields correctly populated

STEP 4 — Browse & Discovery
Build the main tips feed (home screen after login)
Default view: all tips, sorted by upvotes descending
Filter bar: by category, by department, by location
Keyword search across title and content
Tip card design: title, category badge, department, location, semester stamp, upvote count
All tips visible to all logged-in students — no gates
Verify: student can browse, filter, and search all tips immediately after login

STEP 5 — Upvote System
Add upvote button to tip card and tip detail view
One upvote per student per tip (prevent double-voting)
Increment upvotes count in Firestore, add UID to upvotedBy array
Auto-apply verified badge when upvotes cross 10
Verified badge displayed visually on tip card
Verify: upvotes increment correctly, badge appears at threshold, double-voting blocked

STEP 6 — Bookmarks
Add bookmark icon to tip card
Toggle bookmark: add/remove tip ID from user's bookmarks array in Firestore
Build Saved Tips screen (dedicated bookmark view)
Verify: bookmarks persist across sessions, appear in saved screen

STEP 7 — Community Flagging
Add flag/report button to tip card and tip detail view
Increment flagCount in Firestore, add UID to flaggedBy array
When flagCount reaches 3, set hidden: true automatically
Hidden tips do not appear in any feed or search
Verify: tip disappears from feed at 3 flags, reappears if hidden manually reversed by admin

STEP 8 — Ask a Senior Thread
Build dedicated Ask a Senior section
Juniors post questions (title + content)
Any student can reply to questions
Questions displayed with reply count, sorted by most recent activity
Upvote on best answers to surface the most useful response
Verify: question appears, answers thread correctly, upvotes work on answers

STEP 9 — Still Relevant? Button
Add Still Relevant? button to tips older than one semester
Students who find the tip current click to confirm
Display confirmation count on tip card
Verify: confirmation count increments, button only appears on old tips

STEP 10 — Campus Notice Board
Build Campus Notice Board as a separate section
Any logged-in student can post a notice (title + content)
Every notice has a hard expiry of 7 days from creation
Expired notices auto-hidden (Firestore TTL or scheduled function)
Active notices displayed in reverse chronological order
Verify: notice appears, disappears at 7 days, section clearly separate from tips

STEP 11 — Polish, Performance & Responsiveness
Full mobile responsiveness pass — test on common Android screen sizes
Loading states for all data fetches
Empty states (no tips found, no bookmarks yet, etc.)
Error states (failed submission, network error)
Pagination or infinite scroll on tips feed
Performance audit — Firestore query optimisation, index configuration

STEP 12 — Deployment & Pilot Launch
Final QA pass across all flows on mobile and desktop
Configure Firestore security rules for production
Set up Vercel production environment with environment variables
DNS configuration if custom domain is required
Soft launch to a small group (50–100 students) for feedback
Monitor Firebase usage — confirm within free tier limits

Three Layers of Trust
Time Capsule does not rely on a moderation team. The platform is designed so that moderation is largely automatic.

Layer 1 — Verified Identity (Strongest Deterrent)
Every post is tied to a student's institutional email. There is no anonymity. Students do not post explicit or harmful content when their college email is attached to it. This eliminates 90% of the moderation problem before it starts.

Layer 2 — Community Flagging
Three flags from other students auto-hides a post instantly. No admin required. The community self-moderates.

Layer 3 — Admin Deletion
Platform admin (initially the founding team, later a designated college representative) can permanently delete any post directly from the Firebase console. No moderation dashboard required at MVP stage.


Revenue Strategy



Unique Advantages
No platform like this exists — WhatsApp, word of mouth, and handbooks all fail structurally
Verified identity changes behaviour — institutional email removes anonymity and raises content quality automatically
Completely free drives adoption — no friction, no gates, students arrive and immediately find value
Compounding institutional memory — every graduating batch adds tips. Year five is dramatically more valuable than year one
Community-validated quality — upvotes surface the best, flagging removes the worst, the community curates itself
Zero friction access — any browser, any device, one-tap login with an existing Google account
The network effect is geographic — showing a tip to a friend in front of you is the entire go-to-market

Instead of forcing contribution via a paywall mechanic, Time Capsule drives it through identity and social belonging.
